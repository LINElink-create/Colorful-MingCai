# 二、重构优先级

我按“影响稳定性 + 影响开发效率 + 当前代码风险”排序。

---

## P0：必须先做

## 1. 拆分 `observeSelection.ts`

这是当前最优先的技术债。

### 为什么它是 P0

这个文件现在已经混合了：

* selection 监听
* toolbar UI
* translate panel
* note panel
* saved note panel
* 事件注册
* 定位逻辑
* 交互状态机

它已经不是“单文件方便”，而是会直接拖累后续修 bug。

### 拆分目标

建议拆成：

* `selectionState.ts`
* `toolbarController.ts`
* `translationPanelController.ts`
* `notePanelController.ts`
* `savedNotePanelController.ts`
* `selectionPosition.ts`

### 结果要求

`observeSelection.ts` 只保留：

* 初始化
* 依赖注入
* 事件编排

不要再直接写大量 DOM 细节。

---

## 2. 梳理 annotation 创建 / 恢复链路

### 为什么是 P0

这是产品主命脉。

当前关键路径：

* `createAnnotationFromRange`
* `normalizeRange`
* `restoreAnnotation`
* `resolveAnchor`
* `annotationRepository`

这里任何小问题都会导致：

* 用户觉得高亮“不靠谱”
* 笔记附着不稳定
* 页面恢复失败

### 目标

把链路明确拆成三层：

* **创建层**：从 Range 生成 AnnotationRecord
* **恢复层**：从 AnnotationRecord 找回 Range
* **渲染层**：从 Range 生成 mark DOM

避免“创建时直接偷偷走恢复逻辑”的耦合再次混进来。

---

## 3. 统一删除与清空的状态同步模型

### 当前风险

你现在已经支持多入口删除：

* 页面内
* Popup
* 历史总览

这很好，但多入口删除最容易出现：

* UI 已删，存储没删
* 存储已删，页面还留 mark
* 历史页删了，当前页没同步
* 清空后必须 reload 才干净

### 重构目标

建立统一规则：

* 单条删除：优先仓储与 DOM 一起成功
* 批量清空：允许 reload 作为兜底，但要明确
* 跨入口删除：以仓储为真相源，页面端只做同步渲染

---

## P1：第二优先级

## 4. Settings / Popup / History 的状态读取方式统一

你现在已经有多个 composable：

* `usePopupState`
* `useHistoryOverview`
* `useSettingsState`

方向是对的，但要避免每个页面各搞一套消息和刷新模式。

### 目标

统一几个模式：

* 读取当前页数据
* 读取全局 bucket 数据
* 读取设置与 provider 状态
* 监听 storage 变化
* 加载 / 错误状态表现

这样后面改仓储结构时，不会到处修。

---

## 5. 把翻译与核心高亮进一步解耦

### 为什么

当前项目的真正核心是 annotation，不是 translation。

### 目标

做到：

* 翻译失败不影响高亮
* 后端不可用不影响本地功能
* 设置页 provider 异常不影响 popup 核心管理能力

翻译模块应该是附加域，不该污染主链路。

---

## P2：可以跟着做，但不是挡路项

## 6. 样式与组件小幅整理

* 公共按钮样式
* 状态提示样式
* confirm dialog 统一化
* 卡片组件规范

这会提升观感，但不该抢在主链路之前。

---

## 7. 提交记录和版本管理规范化

当前提交历史里已有大量“修复bug / 修改代码 / Version 0.1.2”式提交。

0.2.0 开始建议：

* `feat:`
* `fix:`
* `refactor:`
* `test:`
* `chore:`

这样后面回溯会轻松很多。
